# Feature Specification: 用户登录/注册与离职员工管理

**Feature Branch**: `001-auth-resign-mgmt`
**Created**: 2026-04-08
**Status**: Draft
**Input**: User description: "开发用户登录/注册功能，用户管理，离职员工管理"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用户登录与自注册 (Priority: P1)

新用户需要能够自行注册账号，已有账号的用户能够登录系统。管理员可以创建、管理所有用户账号，普通用户只能查看和修改自己的信息。

**Why this priority**: 认证是系统入口，没有登录/注册就无法使用任何功能。

**Independent Test**: 用户可以通过注册页面创建新账号，使用新账号成功登录，登录后能看到正确的角色权限。

**Acceptance Scenarios**:

1. **Given** 用户未注册，**When** 用户填写注册表单（用户名、密码、显示名称）并提交，**Then** 系统创建账号并自动登录，跳转至首页
2. **Given** 用户已注册，**When** 用户输入正确的用户名和密码，**Then** 系统返回登录成功，跳转至首页
3. **Given** 用户输入错误密码，**When** 用户尝试登录，**Then** 系统提示"用户名或密码错误"
4. **Given** 用户账号被禁用，**When** 用户尝试登录，**Then** 系统提示"账号已被禁用"
5. **Given** 用户已登录，**When** 用户访问登录页面，**Then** 自动跳转至首页

---

### User Story 2 - 用户管理 (Priority: P2)

管理员可以管理系统中的所有用户账号，包括创建子账号、编辑用户信息、重置密码、启用/禁用账号和删除账号。

**Why this priority**: 用户管理是管理员核心功能，依赖登录/注册功能先完成。

**Independent Test**: 管理员可以创建新用户、修改用户状态、重置密码、删除用户，所有操作即时生效。

**Acceptance Scenarios**:

1. **Given** 管理员已登录，**When** 管理员创建新账号（填写用户名、密码、显示名称），**Then** 新账号出现在列表中，状态为启用
2. **Given** 管理员已登录，**When** 管理员禁用某用户账号，**Then** 该用户无法登录系统
3. **Given** 管理员已登录，**When** 管理员重置某用户密码，**Then** 该用户可使用新密码登录
4. **Given** 管理员已登录，**When** 管理员删除非 superadmin 账号，**Then** 该账号从列表中移除
5. **Given** 管理员已登录，**When** 管理员尝试删除 superadmin 账号，**Then** 系统拒绝操作

---

### User Story 3 - 离职员工管理 (Priority: P3)

管理员可以查看所有离职员工信息，按条件筛选搜索，办理员工离职手续，打印离职证明。

**Why this priority**: 离职管理是业务闭环的最后环节，依赖员工基础数据。

**Independent Test**: 管理员可以在在职员工列表中对员工办理离职，离职员工出现在离职列表中，可按条件筛选和打印离职证明。

**Acceptance Scenarios**:

1. **Given** 管理员查看在职员工，**When** 管理员点击"办理离职"并填写离职原因和日期，**Then** 员工状态变为离职，出现在离职列表中
2. **Given** 管理员查看离职员工列表，**When** 管理员按部门/日期/关键词筛选，**Then** 列表显示符合条件的结果
3. **Given** 管理员查看离职员工，**When** 管理员点击"打印证明"，**Then** 系统生成离职证明并可打印
4. **Given** 管理员查看离职员工列表，**When** 管理员翻页或修改每页条数，**Then** 列表正确显示对应数据

---

### Edge Cases

- 用户名已存在时注册失败，提示友好错误信息
- 密码不满足最低要求（至少6位）时，注册/重置密码失败
- Token 过期后，前端自动跳转到登录页
- 离职员工不能再次办理离职
- 并发登录同一账号时，所有 session 保持有效
- 注册的用户默认角色为普通用户（user），需管理员手动提升权限

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供用户注册功能，允许新用户自行创建账号
- **FR-002**: 系统必须提供用户登录功能，验证用户名和密码后签发 Token
- **FR-003**: 系统必须提供"记住我"功能，延长 Token 有效期至 7 天
- **FR-004**: 系统必须提供用户退出功能，清除客户端 Token
- **FR-005**: 管理员必须能够创建、查看、编辑、禁用、删除用户账号
- **FR-006**: 管理员必须能够重置任意用户密码
- **FR-007**: 系统必须禁止删除和禁用 superadmin 账号
- **FR-008**: 系统必须提供离职员工列表，支持按部门、日期范围、关键词筛选
- **FR-009**: 管理员必须能够为在职员工办理离职手续，填写离职原因和日期
- **FR-010**: 系统必须支持打印离职证明
- **FR-011**: Token 过期后系统必须自动跳转至登录页
- **FR-012**: 注册用户默认角色为普通用户（user）

### Key Entities

- **User**: 用户账号，包含用户名、密码（加密）、显示名称、角色（admin/user）、状态（active/disabled）
- **Employee**: 员工信息，包含工号、姓名、部门、职位、手机号、邮箱、入职日期、薪资、状态
- **ResignedEmployee**: 已离职员工（复用 Employee 实体，状态为 resigned），额外包含离职日期、离职原因

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在 30 秒内完成注册流程
- **SC-002**: 用户能在 10 秒内完成登录操作
- **SC-003**: 所有核心功能（登录、注册、用户管理、离职管理）单元测试覆盖率达到 80% 以上
- **SC-004**: 100% 的错误操作都有友好的中文提示信息
- **SC-005**: 离职员工列表在 1000 条数据以内，页面加载时间不超过 2 秒

## Assumptions

- 用户有稳定的网络连接
- 系统仅面向内部员工使用，不需要第三方 OAuth 登录
- 注册不需要邮箱验证
- 离职证明使用浏览器打印功能，不生成 PDF 文件
- 现有的 JWT 认证机制足够满足安全需求
- 已有 superadmin 账号作为初始管理员
